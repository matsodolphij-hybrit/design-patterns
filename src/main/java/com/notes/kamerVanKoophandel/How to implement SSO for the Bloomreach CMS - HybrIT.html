How to implement SSO for the Bloomreach CMS

How to implement SSO for the Bloomreach CMS

This article has been written for developers who work with the Bloomreach Experience manager (formerly known as the Hippo CMS). Developers who want to see a Spring security SSO implementation using Google may find this article useful as well. There are plenty of Spring Security blogs available on the web so why did I write this article? Well… none of them are written in the context of the Bloomreach experience manager. And as we will see shortly there are a few caveats when implementing Spring Security. In this blog we will see how we can implement SSO with OpenID connect for the CMS part of the Bloomreach experience manager.

There will be several references to code in the github repository in order to make this article more readable. Adding all the code in gists is simply too much.

This blog will feature some Spring functionality, but for a more in depth explanation on the key concepts I advise to read the following blog from Baeldung.

1. Getting started

This Spring security implementation has been made in the Bloomreach experience manager version 15.4.0 this means that we will need Java JDK 11 and maven 3.5.0+.

The configuration of the Google Identity provider is not part of this blog, you can find an excellent manual on how to set up your own application within the Google API console. The source code for the SSO implementation can be found in github. It is not required to have a Bloomreach account and it should be possible to run the application out of the box if you have the correct java JDK and maven versions installed. 

2. Implementation

This chapter will take you through each step of implementing Spring Security. The table of contents:

  1. Maven dependencies
  2. Required package structure
  3. Spring properties
  4. Authorization configuration
  5. Login
    1. Custom security provider

In the end we will have replaced to default login screen with a login prompt of our Identity provider (Google). The user will be redirected to the CMS if the login is succesful.

2.1 Maven dependencies

The following dependencies need to be added to the cms-dependencies pom file. It is recommended to use the existing spring-boot.version property from the Bloomreach experience manager. The lombok dependency is optional and can be left out.

2.2 Required package structure

It took me a while to find this caveat. The information was hidden deep within the bowels of the bloomreach documentation platform. Specifically in the upgrade steps manual for version 14.6.0

Note: the exact package “org.bloomreach.xm.cms” must be used! 

This is due to an already existing Spring context which has an auto scan on this package structure. Note that we add this functionality to the cms module and not the cms-dependencies module.

2.3 Spring properties

Add the below properties in the cargo.run maven profile which is located in the root pom of the project. This will ensure that the environment variables for the spring properties are set when developing locally.

<systemProperties>

<!-- OIDC environment variables -->
<sso.enabled>true</sso.enabled>
<spring.security.oauth2.client.provider.google.issuer-uri>https://accounts.google.com</spring.security.oauth2.client.provider.google.issuer-uri>

Then we will add a properties file in the cms module src/main/resources/ folder and name it google-sso.properties. These values are set using environment variables.

2.4 Authorization configuration

The next step is to implement the authorization configuration. This is just a working example of how the configuration can be done. There are four things that are important to note.

The first thing to note is the if statement on line 18. This is a simple feature toggle for the SSO functionality. This way it is possible to switch between SSO login or the default login page.

The second is the .headers().frameOptions().sameOrigin() configuration. The experience manager will break if this is left out.

The third item is the filter that we add after the .oauth2() login configuration. This is our custom Spring filter in which we create the user session. We will use this session later on for creating a new account in the repository.

package org.bloomreach.xm.cms.oidc.config;
import lombok.extern.slf4j.Slf4j;
...
@Slf4j
@Configuration
@EnableWebSecurity
@PropertySource("classpath:google-sso.properties")
public class AuthorizationConfig {
@Value("${sso.enabled}")
boolean ssoEnabled;
@Bean
SecurityFilterChain filterChain(final HttpSecurity http)
throws Exception {
if (ssoEnabled) {
return http
.csrf(AbstractHttpConfigurer::disable)
.headers()
.frameOptions()
.sameOrigin() // this configuration makes sure that the experience manager does not break.
.and()
.authorizeRequests()
.and()
.authorizeRequests(r -> r.anyRequest().authenticated())
.oauth2Login(
oauth2 ->
oauth2.userInfoEndpoint(ep -> ep.oidcUserService(customOidcUserService())))
.addFilterAfter(new UserSessionFilter(), FilterSecurityInterceptor.class)
.build();
} else {
return defaultBRXMLogin(http);
}
}
private OAuth2UserService<OidcUserRequest, OidcUser> customOidcUserService() { .... }
private SecurityFilterChain defaultBRXMLogin(final HttpSecurity http) throws Exception { ... }
}

The fourth item is our customOidcuserService() method. This method is used for extracting the user information. This includes the authorities, the ID of the token and the name and email address of the user. If you use Okta or perhaps Azure, then this is also the place where you can retrieve the users role (admin, editor etc..).

private @NotNull OAuth2UserService<OidcUserRequest, OidcUser> customOidcUserService() {
final OidcUserService delegate = new OidcUserService();
return (userRequest) -> {
OidcUser oidcUser = delegate.loadUser(userRequest);
// Google SSO does not configure authorities. Other providers like Azure AD and OKTA can specify authorities.
// These authorities can be used to set a BRXM role. We will set admin role by default for this User.
final Set<GrantedAuthority> mappedAuthorities = new HashSet<>(oidcUser.getAuthorities());
oidcUser =
new NamedOidcUser(
mappedAuthorities,
oidcUser.getIdToken(),
oidcUser.getUserInfo(),
oidcUser.getEmail());
return oidcUser;
};
}

2.5 login

The previous step defined a custom Spring filter which is set to apply after the oAuth2 (OpenID Connect flow) . We will take a closer look at this filter in this chapter.

The filter has the following responsibilities.

  • The creation of a simple credentials object containing the user info and the ID of the token.
  • It creates a new HTTP Session with the ID of the token as identifier.
  • It adds the simple credentials object to the newly created HTTP Session.
package org.bloomreach.xm.cms.oidc.filter;
import ...
@Slf4j
public class UserSessionFilter implements Filter {
public static final String FIRSTNAME_ATTRIBUTE = "firstname";
public static final String LASTNAME_ATTRIBUTE = "lastname";
public static final String EMAIL_ATTRIBUTE = "email";
private static final String SSO_USER_STATE = SSOUserState.class.getName();
private static final ThreadLocal<SSOUserState> tlCurrentSSOUserState = new ThreadLocal<>();
@Override
public void init(FilterConfig filterConfig) { }
/**
* Creates a new secured session if the user is authorized. {@inheritDoc}
*/
@Override
public void doFilter(
final ServletRequest servletRequest,
final ServletResponse servletResponse,
final FilterChain filterChain)
throws IOException, ServletException {
final Authentication authentication = SecurityContextHolder.getContext().getAuthentication();
if (!authentication.isAuthenticated()) {
filterChain.doFilter(servletRequest, servletResponse);
}
final HttpSession session = ((HttpServletRequest) servletRequest).getSession();
SSOUserState userState = (SSOUserState) session.getAttribute(SSO_USER_STATE);
if (userState == null || !userState.getSessionId().equals(session.getId())) {
userState = new SSOUserState(new UserCredentials(createSimpleCredentials(authentication)), session.getId());
session.setAttribute(SSO_USER_STATE, userState);
}
if (userState.getSessionId().equals(session.getId())) {
servletRequest.setAttribute(UserCredentials.class.getName(), userState.getCredentials());
}
try {
tlCurrentSSOUserState.set(userState);
filterChain.doFilter(servletRequest, servletResponse);
} finally {
tlCurrentSSOUserState.remove();
}
}
public static SSOUserState getCurrentSSOUserState() { ... }
@Override
public void destroy() { ... }
private SimpleCredentials createSimpleCredentials(final Authentication authentication) {
final NamedOidcUser userInfo = (NamedOidcUser) authentication.getPrincipal();
final SimpleCredentials simpleCredentials =
new SimpleCredentials(userInfo.getEmail(), "dummy".toCharArray());
simpleCredentials.setAttribute(FIRSTNAME_ATTRIBUTE, userInfo.getGivenName());
simpleCredentials.setAttribute(LASTNAME_ATTRIBUTE, getFullSurname(userInfo));
simpleCredentials.setAttribute(EMAIL_ATTRIBUTE, userInfo.getEmail());
simpleCredentials.setAttribute(SSOUserState.OIDC_ID, userInfo.getEmail());
return simpleCredentials;
}
private String getFullSurname(final NamedOidcUser userInfo) { ... }

For the java implementation of this security provider you will need to add a class which extends the DelegatingSecurityProvider class and override its methods. In these methods we will retrieve the HTTP Session which contains the user information. The validation is done by checking that the user information is not empty. The HTTP Session can only contain data if the user has a successful login at Google.

package org.hybrit.repository.provider;
import ...
@Slf4j
public class OpenIdDelegatingSecurityProvider extends DelegatingSecurityProvider {
private HippoUserManager userManager;
public OpenIdDelegatingSecurityProvider() {
super(new RepositorySecurityProvider());
}
@Override
public UserManager getUserManager() throws RepositoryException {
if (userManager == null) {
userManager =
new DelegatingHippoUserManager((HippoUserManager) super.getUserManager()) {
@Override
public boolean authenticate(final SimpleCredentials simpleCredentials)
throws RepositoryException {
if (validateAuthentication(simpleCredentials)) {
final String userId = simpleCredentials.getUserID();
if (!hasUser(userId)) {
// This where we create a new user with a role of admin. This is where you can set another role if your SSO provider allows this.
syncUser(
createUser(userId), getGroupManager().getGroup("admin"), simpleCredentials);
}
return true;
} else {
return false;
}
}
};
}
return userManager;
}
@Override
public UserManager getUserManager(Session session) throws RepositoryException { ... }
protected boolean validateAuthentication(final SimpleCredentials simpleCredentials) {
final SSOUserState userState = UserSessionFilter.getCurrentSSOUserState();
if (userState != null) {
return StringUtils.isNotEmpty(userState.getCredentials().getUsername());
} else {
final String oidcID = (String) simpleCredentials.getAttribute(OIDC_ID);
if (StringUtils.isNotBlank(oidcID)) {
log.debug("Authentication allowed to: {}", oidcID);
return true;
}
}
return false;
}
protected void syncUser(final Node user, final Node group, final SimpleCredentials credentials)
throws RepositoryException {
// By setting this property we make sure that this user will be logged in using our custom OpenIdDelegatingSecurityProvider class.
user.setProperty(
"hipposys:securityprovider",
"openid");
user.setProperty("hipposys:active", true);
user.setProperty("hipposys:firstname", (String) credentials.getAttribute(FIRSTNAME_ATTRIBUTE));
user.setProperty("hipposys:lastname", (String) credentials.getAttribute(LASTNAME_ATTRIBUTE));
user.setProperty("hipposys:email", (String) credentials.getAttribute(EMAIL_ATTRIBUTE));
// add the user to the correct group based on the role
final Value[] values = group.getProperties("hipposys:members").nextProperty().getValues();
final Value[] newValues = Arrays.copyOf(values, values.length + 1);
newValues[values.length] = new StringValue(user.getName());
group.setProperty("hipposys:members", newValues);
}
}

The custom security provider upserts authorized users in the repository and thereby giving the users a CMS account which could include a role if your organization requires it. The default role for each CMS user in our case has Admin privileges.

3. Summary

We have taken away the usermanagement from the Bloomreach experience manager and moved it to Google. This means that new CMS users need to be authorized in the Identity Provider, this includes removing login access or changing roles.

Things to take into consideration are:

  1. User accounts are not removed automatically. If your implementation is hosted in Mission control you may need to have a cronjob for removing accounts due to a limit in the number of user accounts allowed. This can be done safely due to the upsert mechanism in the security provider at login time.
  2. Navigating to ?/cms prompts the login screen of Google. There is no longer a Bloomreach experience manager login screen.
  3. The user navigates to ?/cms after a logout, if the user still has login priveliges then an automatic login follows. It should be possible to add a custom logout url to prevent the user from automatically logging in again
